What is Wonder

Wonder stands for Workflow on Distributed Environment. It's a workflow management system architecture provides a highly distributed framework to support Workflow Execution in large-scale environments. It uses the mobile objects paradigm, together with the communication facilities and resources of the CORBA framework. This project was part of a MS Thesis project of Roberto Silveira Silva FIlho at the IC - Institute of Computing at UNICAMP - University of Campinas - SP - Brazil. The dissertation was defended on July 2000. It is available here.

Wonder Architecture

The main components and issues of the Wonder architecture are described in the paper published in the 4th International Symposium on Autonomous Decentralized Systems (ISADS '99), March 20-23, 1999 - Tokyo, JAPAN - CORBA Based Architecture for Large Scale Workflow.

It is also summarized in the poster (in portuguese) presented in the 17th Brazilian Symposium on Computer Networks - (SBRC'99) -Bahia - Brazil - May 25-28, 1999. - Wonder: A Distributed Architecture for Large Scale Workflow Using CORBA.

An updated version of the architecture, with some implementation issues is in the paper, published in the IEEE/IEICE Special Issue on Autonomous Decentralized Systems of the IEICE Transactions on Communications magazine, Japan, June 2000.

The tests and the final implementation version is described in the Master Thesis (in portuguese): Uma Arquitetura Baseada em CORBA para Workflow de Larga Escala.

A six-page paper with a brief description of the architecture and the test results of the architecture is also available. Submitted to the CTD 2001 - XIV Tesis and Discertation Workshop of the SBC - Brazilian Computer Society.

The main components of the Wonder architecture are summarized below:

  • Activity Manager - Mobile agent that conveys all case data to the actor's host. It implements the workflow engine which allows its authonomous migration. It also provides the data and case mobility of the architecture.
  • Wrapper - This component is used by the Activity Manager to invoke external applications as word processors, CAD tools, spreadsheets, and so on. It executes and manages these Invoked applications during a Task execution.
  • Synchronization Activity - Implemets the synchronization points (AND and OR joins), udsed do synchronize and merge data from workflow branches.
  • Case Coordinator - Corodinates and manages a case instance executuion. Tracks the execution of the AcrivityManager(s) agents of a particular workflow case. It provides fault tolerance and management facilities to the architecture.
  • Process Coordinator - Creates and manages Case Coordinators for each case instance. Copes with scalability and performs the case coordinator creation and failure recovery procedure.
  • Role Coordinator - Dynamically allocates actors to perform activities. Helps in the ActivityManager migration and in the dynamic binding of actors and resources in the system.
  • Backup Server - Stores data used during the activities execution. Used for improved fault tolerance. It copies critical data from unreliable desktops to more stable machines. Also provides data to fault recovery procedures.
  • History Server - Stores the execution history of each finalized case. Provides a database front-end used by the role coordinators to dynamically alocate users based on previou execution history.
  • Task List - User takslit surrogate object. Implements the communication between an actor and the WONDER system.

The object persistence is implemented by the LOA - Local Object Activator - A "hook" in the orbixd daemon, a Locator that executes in each host of the system. The objects are stored in the Object Repository, a special local disk area.

Project Status

A prototype of the architecture with the following components were implemented for my MS thesis. Its architecture, implementation and tests are described in my dissertation (in portuguese).

  • LOA: The Local Object Activator that implements the persistence of the Wonder Objects;
  • LockManager: A concurrency monitor for the LOA: Reading during object serialization must be avoided. It is not used anymore by the system.
  • Process Interpreter: The parser that reads a PLISP plan and interprets. It is used by the Coordinators and The Activities. The Process Definition (Workflow) is described in PLISP (Process Lisp) a Lisp like language specially developed for this project;
  • Activity Manager: The Object that implements the migration and execution of activities. An optimized data migration process was implemented. It transfers not necessary data as links that are further converted to files (fetched from the previous owner) when needed.
  • Wrapper: The Object that manages the execution of an invoked application and notifies the ActivityManager when it is finished.
  • Synchronization Activity: Object that implements and-joins/or-joins.
  • Process Coordinator: Creates and manages the cases.
  • Case Coordinator: Manages a case instance and its activities. Performs the garbage collection by the end of the case, removing the object state and data from the system hosts.
  • Role Coordinator: Implements a "yellow page" service of uses and roles.
  • Task List: The object that stores information and activities that the user is executing.

The migration mechanism is completely functional, including the data/links transfer optimization police.
A language for definition and creation of a test environment was also implemented. It is interpreted by the Wstarter application that creates the initial system object configuration.

The system was tested using concurrent processes in centralized and distributed scenarios. The tests can be found at the Performance Tests page.
