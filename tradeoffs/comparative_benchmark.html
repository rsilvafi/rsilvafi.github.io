Overview

After performing individual benchmarks with different infrastructures and applications, we compared the results obtained in these experiments in a horizontal way.

It is important to note that infrastructures as Siena, CORBA-NS and Java Spaces are reused "from the outside". I.e. they are extended as black boxes, by building the required functionality around their provided APIs. YANCEES, on the other hand, is configurable and extensible as a grey box, allowing the modification of the set of features its supports "from the inside". Performance: usefulness and efficiency

The performance tests demonstrate: (1) The usefulness of YANCEES and the other infrastructures in supporting different applicaiton domains; and (2) the efficiency of these infrastructures when reused in different contexts.

EDEM performance results:


In EDEM benchmarks, both JavaSpaces and CORBA-NS were not able to handle the large amount/frequency of events demanded by the benchmark. Moreover, these infrastructures must be executed in different processes.

CASSIUS performance results:


In the CASSIUS scenario, the native pull model of JavaSpaces gives it a performance advantage. Whereas all the other infrastructures, including CORBA-NS, need to implement a push-based event queue that store notifications for later retrieval. This strategy results in higher processing times for the other infrastructures.

IMPROMPTU performance results:


In the IMPROMPTU scenarion, the JavaSpaces pull notificaiton model does not perform well in this intrinsic push, fast forwarding application domain.

Summary of Findings

Performance

- The performance of compositional (YANCEES) and minimal-core (Siena) infrastructures are comparable to the best case build-from-scratch approach implementations.

- Generalization versus specificity

- Generalization (in either subscription and notificatoin models), while support a larger set of application domains, require extra programming effort to translate and parse specific concerns into generalized data and subscription formats.

- Reuse: buy versus build

- The ability to customize the API to match application requirements (YANCEES case) has two benefits:
  1. it divides the development effort between infrastructure producers and consumers.
  2. it reduces client-side development effort since the infrastructure can be customized to support the exact set of features required by the domain.
- However, these benefits come with high initial software development costs. In order to pay-off that initial investment, YANCEES (as well as the other approaches) need to be reused in different contexts.

- The costs of inflexibility

- The inflexibility of CORBA-NS, Siena and JavaSpaces requires API mismatches to be handled in the client-side, by the application developers. This not only increases the development effort, but also impacts the performance of the application.

Examples of mismatches found during the benchmark:
    - JavaSpaces inability to handle aritmetic filter operators as: <, >, <=, >=
    - JavaSpaces lack of support for pull notifications
    - JavaSpaces requires clients to find the tuple when a notification trigger is fired.
    - CORBA-NS pull notifications does not match CASSIUS pull model
    - Siena does not support pull notifications
    - Siena Pattern implementation is different from the one required by EDEM.

- Variation-based approaches as CORBA-NS provide a large set of options in its API. The selection between these options needs to be performed manually. The consequence is an increase in the infrastructure consumers' development effort.
- The JavaSpaces model does not completely match the publish/subscribe model, requiring extra effort to retrieve the events (represented as tuples), from the space. This results in performance penalties and extra development effort.
